# Evaluation Rubric

Use this rubric to score Skills from a product-design perspective. Give each dimension a 1-5 score and cite evidence.

## 1. 触发质量

- 5: `name` 短小准确，`description` 清楚说明何时使用，并覆盖用户真实说法。
- 3: 名称可识别，但触发语偏抽象或覆盖不完整。
- 1: 名称泛化，description 只说能力，不说明触发场景。

Check:
- 是否避免 `pm-helper`、`assistant` 这类泛化名称。
- 是否覆盖真实请求，例如“帮我写 PRD”“整理需求池”“评价这个 skill”。

## 2. 流程质量

- 5: 有明确先后顺序、判断点、异常处理和退出条件。
- 3: 有步骤，但关键分支和异常处理不足。
- 1: 只有原则列表，缺少可执行流程。

Check:
- 是否体现资深执行者的工作顺序。
- 是否处理信息不完整、口径不清、需求冲突、目标路径缺失等异常。

## 3. 输出质量

- 5: 输出结构稳定，可直接进入评审、研发、测试、汇报或后续执行。
- 3: 有格式要求，但缺少完整模板或证据字段。
- 1: 主要是漂亮文字，缺少决策价值。

Check:
- 是否保留证据、假设、不确定性和下一步动作。
- 是否有固定章节、表格或可复制结构。

## 4. 经验密度

- 5: 包含普通模型未必知道的组织经验、业务规则、反例、常见坑和样例。
- 3: 有一些方法论，但缺少具体好坏标准。
- 1: 只是说明“做什么”，没有说明“怎么做得好”。

Check:
- 是否提供好输出/坏输出对比。
- 是否有审查清单帮助 Agent 自检。

## 5. 资源组织质量

- 5: `SKILL.md` 精简，细则、模板、案例放入 `references/`，确定性任务放入 `scripts/`。
- 3: 单文件可用，但有膨胀趋势。
- 1: 大量模板、规范、样例堆在 `SKILL.md`，上下文成本高。

Check:
- `SKILL.md` 是否尽量低于 5k 字。
- assets 是否只放输出会直接使用的模板、图片、样板文件。

## 6. 自动化质量

- 5: 脚本可无上下文执行，输入输出清晰，能处理错误输入，并提供示例命令。
- 3: 有脚本，但接口、错误处理或输出格式不够稳定。
- 1: 每次都让 Agent 重新发明数据清洗、扫描或格式转换逻辑。

Check:
- 脚本是否支持机器可读输出。
- 错误是否清楚，退出码是否可靠。

## 7. 可维护性

- 5: 模板、指标、业务规则容易更新，信息没有重复维护，能沉淀真实失败案例。
- 3: 文件结构清楚，但缺少迭代规则。
- 1: 规则散落或重复，后续维护容易漂移。

Check:
- references 是否职责分明。
- 是否说明真实项目后如何更新 Skill。

## 8. 可评测性

- 5: 有测试样例、好坏输出样例、checklist 和失败案例记录机制。
- 3: 有少量自检规则，但不能稳定评审输出质量。
- 1: 没有评测样例，也没有完成前检查。

Check:
- 是否能对输出做 checklist 评审。
- 是否能把失败案例反向更新到 Skill。
